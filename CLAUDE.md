# ProjectBM - BM 기획자 시뮬레이터

## 프로젝트 개요
BM(Business Model) 기획자가 상급자 보고 및 포트폴리오용으로 제작하는 캐릭터 성장 시뮬레이터.

## 배경 지식

### 비즈니스 구조
- 패키지 상품을 판매함
- 패키지 상품에는 다수의 **소환권**이 포함됨
- 소환권 사용 시 캐릭터가 성장함 (캐릭터 등급, 펫 등급 등)

### 핵심 개념 (예시)
- 불멸 클래스 (Immortal Class): 캐릭터 등급 중 하나
- 불멸 펫 (Immortal Pet): 펫 등급 중 하나
- 달성률: 특정 목표 스펙 대비 현재 보유 비율 (예: 불멸 클래스 달성률 90%)

## 원하는 기능

### 입력
1. **캐릭터 목표 스펙 입력**: 불멸 클래스 몇 개, 불멸 펫 몇 개 등 목표치 설정
2. **판매 패키지 입력**: 판매한 패키지 종류 및 수량 (내부 소환권 수량 포함)

### 출력/시각화
- 시간 흐름에 따른 캐릭터 성장 시각화
- 월별 달성률 변화 표시 (예: 11월 불멸클래스 달성률 90% → 12월 99%)
- 보기 좋고 아름다운 UI (상급자 보고 + 포트폴리오 목적)

## 방향성
- 시각적으로 예쁘고 인상적인 UI/UX
- 상급자 보고용: 직관적이고 명확한 데이터 전달
- 포트폴리오용: 전문성과 완성도

## 개발 원칙

### 질문 우선
- 세부 사항이 불명확할 경우 **반드시 사용자에게 질문하여** 더 나은 기획을 만들 것
- 새로운 기능이나 데이터 구조 설계 전, 필요한 정보를 먼저 파악할 것
- 게임 도메인 용어나 BM 구조에 대해 불확실하면 물어볼 것

### 수정 용이성 (매우 중요)
이 프로젝트는 진행 중 **잦은 수정이 예상**된다. 사용자의 기획 변경, AI 오류 수정 등 다양한 이유로 코드가 계속 바뀔 수 있음.

- **확률/규칙 값은 반드시 상수로 분리**: 로직 안에 숫자를 직접 넣지 않고 설정 파일이나 상수 블록에 모아둘 것
- **데이터와 로직을 분리**: 패키지 데이터, 소환권 확률 등은 별도 데이터 파일로 관리
- **컴포넌트/모듈 단위로 쪼개기**: 기능 하나가 파일 하나. 수정 시 다른 부분에 영향 최소화
- **변경 이력 추적**: 중요한 수치나 로직 변경 시 작업로그에 반드시 기록
- **하드코딩 금지**: 나중에 바꿀 가능성이 있는 값(확률, 개수, 등급명 등)은 절대 코드 안에 직접 쓰지 말 것

## 세션 로그 프로토콜

### 작업 시작 시
- `작업로그/` 디렉토리의 최근 로그 파일들이 자동으로 컨텍스트에 주입됨 (훅)
- 로그를 숙지한 후 작업에 들어갈 것
- 미결 사항이 있으면 작업 전 사용자에게 먼저 확인할 것

### 세션 종료 시 (반드시 수행)
1. `작업로그/YYYY-MM-DD.md` 파일 생성 또는 업데이트
2. 로그에 포함할 내용:
   - 오늘 논의된 내용 요약
   - 결정된 사항
   - 완료된 작업
   - 미결 사항 / 다음 세션에서 할 일
3. git commit & push

### 로그 파일 네이밍
- `작업로그/YYYY-MM-DD.md` 형식
- 같은 날 여러 세션이면 동일 파일에 누적 작성

## 개발 작업 체크리스트

### 매 세션 시작 시 반드시 수행
1. 작업로그 자동 주입 내용 확인 (훅이 삽입한 <작업_로그> 태그 내용 숙지)
2. 미결 사항 있으면 코드 작업 전 사용자에게 먼저 확인
3. 오늘 할 작업 범위 사용자와 합의 후 시작

### 코드 작성 시 반드시 지킬 것
-  에 있는 상수/데이터 파일을 수정할 일이 있으면, 로직 코드는 건드리지 말 것
- 컴포넌트 하나가 200줄 넘어가면 분리를 검토할 것
- 새 파일 만들기 전에 기존 파일에 넣을 수 있는지 먼저 검토
- PR/커밋 단위: 기능 하나 = 커밋 하나 (너무 크게 묶지 말 것)

### 수정 요청이 들어왔을 때
1. 변경 범위가 데이터인지 / 로직인지 / UI인지 먼저 파악
2. 데이터 변경 →  파일만 수정
3. 로직 변경 →  파일만 수정
4. UI 변경 →  파일만 수정
5. 변경 후 작업로그에 무엇을 왜 바꿨는지 기록
