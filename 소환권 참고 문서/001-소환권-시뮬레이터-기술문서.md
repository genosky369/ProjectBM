## 개요

소환권 시뮬레이터는 두 개의 파일로 운영됩니다.

| 파일 | 설명 | 기능 범위 |
|------|------|-----------|
| `소환권 기댓값 계산기/app.py` | 독립 실행 버전 (v4.6) | 풀 기능 (3가지 모드) |
| `modules/summon_calculator.py` | BM 통합 웹 통합 모듈 | 간소화 버전 (1가지 모드) |

> **주의**: 두 파일은 코드가 유사하지만 독립적으로 관리됩니다. 통합 모듈(`summon_calculator.py`)은 독립 앱(`app.py`)의 일부 기능만 포함합니다.

## 핵심 데이터 구조

### 등급 체계 (CANON_GRADES)

7단계 등급, 낮은 순서부터:

```
일반 → 고급 → 희귀 → 영웅 → 고대 → 전설 → 불멸
```

인덱스로도 접근 가능 (`GRADE_INDEX`): `{"일반": 0, "고급": 1, ... "불멸": 6}`

### 시스템 종류 (SYSTEMS)

4가지: `클래스`, `펫`, `투혼`, `카드`

각 시스템의 합성 최대 등급:

| 시스템 | 최대 등급 | 특이사항 |
|--------|----------|---------|
| 클래스 | 불멸 | 불멸은 합성 불가, 승급식으로만 획득 |
| 펫 | 불멸 | 전설→불멸 합성 가능 |
| 투혼 | 전설 | |
| 카드 | 전설 | |

### Ticket 클래스

```python
@dataclass
class Ticket:
    system: str            # "클래스" / "펫" / "투혼" / "카드"
    name: str              # 티켓 이름 (UI에서 표시)
    probs: Dict[str, float]  # 등급별 확률 (합계 1.0)
    pulls: int             # 사용 1회당 뽑기 수 (11회권=11, 도전/확정=1)
```

### 티켓 종류

각 시스템에 세 가지 종류의 티켓이 있습니다.

**11회 소환권** (pulls=11): 한 번 사용 시 11번 뽑기, 등급별 확률로 결과 결정
- 찬란한: 일반/고급 위주 (일반 70%, 고급 27.6%, ...)
- 신비로운: 고급 위주 (고급 97.8%, ...)
- 눈부신: 희귀 위주 (희귀 98.9%, ...)
- 영롱한: 희귀+영웅 (희귀 90%, 영웅 9%, ...) — 클래스/펫만 존재

**도전 소환권** (pulls=1): 1번 뽑기, 특정 등급 이상 확률
- 영웅 도전: 희귀 95% / 영웅 5%
- 고대 도전: 영웅 86~88% / 고대 12~14% (시스템별 차이)
- 전설 도전: 고대 90% / 전설 10%

**확정 소환권** (pulls=1): 100% 특정 등급 확정
- 영웅 확정 / 고대 확정 / 전설 확정

> `make_ticket()` 함수는 입력값을 자동으로 정규화(합계 1.0으로)하므로 확률을 퍼센트(%)로 입력해도 됩니다.

## 합성 시스템

### 합성 규칙 (SYNTH_RULES)

형식: `(현재 등급, 상위 등급, 필요 개수, 성공 확률, 실패 반환, 천장 N회)`

**클래스 합성 규칙**:
| 현재 | 상위 | 필요 | 성공률 | 실패 반환 | 천장 |
|------|------|------|--------|-----------|------|
| 일반 | 고급 | 4개 | 25% | 1개 | 없음 |
| 고급 | 희귀 | 4개 | 20% | 1개 | 없음 |
| 희귀 | 영웅 | 4개 | 18% | 1개 | 20회 |
| 영웅 | 고대 | 4개 | 14% | 1개 | 15회 |
| 고대 | 전설 | 3개 | 10% | 1개 | 10회 |

> 실패 시 재료 1개 반환 = 사실상 3개 소모 (4-1=3), 다음 시도에도 재사용

**천장(Pity)**: N회 연속 실패 시 무조건 합성 성공. 성공 후 pity 카운터 초기화.

### SynthesisState 클래스

```python
@dataclass
class SynthesisState:
    pity: Dict[Tuple[str, str], int]  # (현재 등급, 상위 등급) → 연속 실패 횟수
```

시뮬레이션 런마다 새로 생성. 합성 상태(천장 카운터)를 추적합니다.

### try_synthesis() 함수

```
입력: system, inv(인벤토리), state(합성 상태), max_grade(합성 상한 등급)
출력: 없음 (inv를 직접 수정)
```

동작 방식:
1. 낮은 등급(일반)부터 순서대로 순회
2. 재료가 충분하면(`inv[frm] >= need`) 합성 시도
3. 성공: 상위 등급 +1, pity 초기화
4. 실패: 재료 1개 반환, pity +1 → pity가 천장에 도달하면 강제 성공
5. 더 이상 합성 가능한 것이 없을 때까지 반복 (`while True` 루프)
6. `max_grade` 설정 시 해당 등급 초과 합성은 건너뜀

## 시뮬레이션 모드 (독립 앱 v4.6 기준)

### 모드 1: 티켓 묶음 기대값 계산기

**합성 전 기대값** (`bundle_expected_pre`)

- **성격**: 순수 수학 계산 (랜덤성 없음, 즉시 실행)
- **계산식**: `기대 개수[등급] = 티켓별 (pulls × count × 등급 확률)`의 합산
- **출력**: 등급별 기대 개수 (소수점)

**합성 후 기대값** (`bundle_expected_post`)

- **성격**: Monte Carlo 시뮬레이션 (N번 반복, 수십 초 소요)
- **동작**:
  1. 각 반복(run)마다: 모든 티켓을 지정 수량만큼 뽑기 실행 → `draw_from_ticket()`
  2. 뽑기 결과 인벤토리에 합산
  3. `try_synthesis()`로 합성 적용
  4. 최종 인벤토리 기록
- **출력**: 평균, P50(중간값), P90 (각 등급별)
- **클래스 불멸 참고값**: 시뮬레이션 결과의 영웅/고대/전설 수량으로 만들 수 있는 불멸 개수 계산

```
불멸 1개 승급식: 영웅 50개 + 고대 16개 + 전설 4개 + 응축된 각성의 기운 2735개 + 불멸의 기운 3개
```

### 모드 2: 결과 기여 비중(%) 기반 필요 장수 산정 (v4.6 신규)

목표: "전설 8개를 얻으려면 각 티켓이 몇 장씩 필요한가?" 를 역산

**동작 과정**:
1. 각 티켓별 기여 비중(%)을 입력 (합계 100%)
2. `micro_expected_target_per_ticket()`으로 티켓 1장당 목표 등급 생산 기대값(e_i) 추정
   - pack_size장을 연속 사용하는 trial을 N번 반복 (피티 누적 반영)
   - 확정권은 e_i = 1.0으로 처리
3. 역산: `필요 장수 = ceil(목표 개수 × 기여 비중 / e_i)`
4. 검증: 산정 장수로 실제 시뮬레이션 실행 → 실제 결과와 비교

**주요 파라미터**:
- `micro_runs`: e_i 추정 trial 반복 수 (많을수록 정확, 느림)
- `pack_size`: 한 trial에서 연속 사용 장수 (피티 누적 반영 수준)

### 모드 3: 퍼센트(사용 장수 비율) 시뮬레이션

목표: "이 티켓 조합으로 목표를 달성하려면 총 몇 장이 필요한가?"

**동작 과정**:
1. 각 티켓별 사용 비율(%)을 입력 (합계 100%)
2. SWRR 방식으로 티켓 순서 결정 후 한 장씩 사용
3. 매 장 사용 후 합성 적용 → 목표 달성 여부 확인
4. N번 반복 → 통계 집계

**SWRR (Smoothed Weighted Round-Robin)**:
```python
scores = (w * total_used) - used_counts
idx = int(np.argmax(scores))
```
`w`: 각 티켓의 목표 비율, `used_counts`: 현재까지 사용 횟수
→ 비율에 맞게 균등하게 티켓을 섞어 사용 (단순 확률 샘플링보다 비율 편차가 적음)

**출력**:
- 성공률 (목표 달성 비율)
- 필요 장수: 평균, P50, P90, P95
- 티켓별 사용량 분해

## 코드 작업 시 유의사항

### 티켓 확률 수정
`TICKETS` 리스트에서 `make_ticket()` 또는 `Ticket()` 생성자 수정.
- `make_ticket()` 사용 시: 확률을 퍼센트(%)로 입력, 자동 정규화
- `Ticket()` 직접 사용 시: `probs` 딕셔너리의 합계가 1.0이어야 함

### 합성 규칙 수정
`SYNTH_RULES` 딕셔너리 수정. 형식:
```python
("현재_등급", "상위_등급", 필요_개수, 성공_확률, 실패_반환_개수, 천장_N회)
# 천장 없으면 0
```

### 새 시스템/등급 추가
1. `CANON_GRADES` 리스트에 등급 추가 (순서 중요)
2. `SYSTEMS` 리스트에 시스템 추가
3. `SYSTEM_MAX_GRADE`에 최대 등급 등록
4. `SYNTH_RULES`에 합성 규칙 추가
5. `TICKETS` 리스트에 해당 시스템 티켓 추가
6. `CLASS_IMMORTAL_RECIPE` 같이 특수 로직이 있다면 별도 추가

### 두 파일 간 동기화
독립 앱(`app.py`)에서 기능을 추가/수정한 경우, 통합 모듈(`summon_calculator.py`)에도 반영이 필요한지 검토 필요.
현재 통합 모듈은 의도적으로 간소화되어 있으므로 무조건 동기화할 필요는 없음.

## 비고
- 시뮬레이션은 `numpy` 난수 기반이므로 시드 고정 시 재현 가능
- 클래스 불멸은 합성 경로가 없으며 승급식(재료 조합)으로만 획득
  - 시뮬레이션 결과에서 클래스 불멸 관련 수치는 "참고값"임
- P50 = 중간값(절반은 이 수치보다 적게 필요), P90 = 90%의 사람이 이 수치 이하로 달성 가능
